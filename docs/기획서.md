## 1. 문서 개요

### 1-1. 문서 목적
- 감사·결산 실무의 반복 업무(자료 수집·정리·검증·증감분석·문서화)를 대상으로, 결정적 코드와 LLM의 역할을 분리한 지원 도구 "AuditPilot"의 PoC 기획 내용을 정의한다.
- 업무 분석(As-Is) → 과제 도출 → 요구사항 정의 → 시스템 구성 → 일정·리스크의 순서로 기술하며, PoC 범위와 이후 로드맵을 구분하여 확정한다.

### 1-2. 대상 독자
- 1차: 본 도구를 결산 검토·세무조정·소규모 감사 지원 업무에 직접 사용할 개업 회계사 (작성자 본인 포함)
- 2차: 회계법인 단위 도입 검토 시의 품질관리·감사방법론 담당자, 개발 협력자

### 1-3. 용어 정의

| 용어 | 정의 |
|---|---|
| PBC (Prepared by Client) | 감사인이 회사에 요청하여 수취하는 자료 일체. 원장, 시산표, 계정명세서, 계약서 등 |
| 분석적검토 (분석적절차, Analytical Procedures) | 감사기준서 520의 공식 용어는 '분석적절차'이며 본 문서의 '분석적검토'는 그 실무 관행 명칭이다. 재무 데이터 간 관계 분석을 통해 비경상적 변동·이상 징후를 식별하는 감사절차로, 전기 대비 증감분석이 대표적 |
| 감사주장 (Assertion) | 재무제표에 내재된 경영진의 주장. 거래유형·사건에 대한 주장(발생사실·완전성·정확성·기간귀속·분류)과 기말 계정잔액에 대한 주장(실재성·권리와 의무·완전성·정확성평가와 배분)으로 구분된다(감사기준서 315) |
| 시산표 (Trial Balance) | 총계정원장 잔액을 계정별로 집계한 표. 명세서·원장 대사의 기준점 |
| 계정명세서 | 특정 계정 잔액의 세부 구성내역(거래처별·항목별). 시산표 잔액과의 합계 대사가 수취 직후 첫 검증 대상 |
| 조서 (Workpaper) | 수행한 감사절차의 성격·시기·범위, 입수한 감사증거, 도달한 결론의 기록(감사기준서 230). 감사의견의 근거이자 감사가 감사기준에 따라 계획·수행되었다는 증거 |
| 중요성 금액 (Materiality) | 왜곡표시가 재무제표 이용자의 의사결정에 영향을 미치는지 판단하는 금액 기준. 개별 항목 스크리닝에는 수행중요성(Performance Materiality)·명백히 사소한 왜곡표시 기준금액이 함께 사용된다 |
| HITL (Human-in-the-Loop) | 자동화 파이프라인 내에 인간(감사인)의 검토·승인 지점을 구조적으로 배치하는 설계 방식 |

## 2. 추진 배경

감사 실무는 "자료를 받고 → 맞는지 확인하고 → 엑셀을 가공하고 → 전기 조서와 비교하고 → 숫자가 왜 달라졌는지 확인하고 → 회사에 다시 물어보고 → 그 내용을 조서에 남기는" 루프의 반복이다. 감사인의 판단은 이 업무의 핵심이지만, 판단에 도달하기 전까지 거쳐야 하는 반복 작업의 양이 판단 자체보다 훨씬 크다는 것이 실무 경험에서 얻은 문제의식이다.

- **분석적검토**: 전기 대비 증감이 크면 "증가했습니다"로 끝나지 않는다. 계정명세서·원장·회사 설명·증빙을 오가며 증가가 어떤 성격(Nature)에서 발생했는지, 어느 요인이 큰지, 회사 설명이 숫자와 정합한지를 확인해야 한다. 이 과정의 상당 부분은 데이터 집계·대사·재가공이다.
- **주석검토**: 주석 backdata가 재무제표 숫자와 맞는지, 주석 간 참조가 맞는지, 전기 대비 누락된 공시가 없는지 — 숫자 하나하나가 전부 이어져 있고, 그 연결을 사람이 눈으로 좇는다.
- **조회서 대사**: 금융기관 회신을 회사제시자료와 대사하고, 차이가 나면 유선으로 확인하고 다시 정리한다. 여러 회사·여러 기관에 반복된다.

이 경험에서 나온 질문은 세 가지다. ① 자료 형태만 표준화되어 있어도 훨씬 빨리 볼 수 있지 않은가. ② 전기 대비 변동과 이상치를 시스템이 먼저 잡아주는 구조를 만들 수 없는가. ③ 조서 초안과 증감분석 문장의 틀이라도 자동으로 잡아주면 시간이 얼마나 줄어드는가.

AuditPilot의 출발점은 거창한 미래상이 아니라, 지금 감사·결산 현장에서 실제로 시간이 들어가는 지점을 줄이는 것이다. 단, 핵심 전제가 있다. **반복 작업을 줄이되, 감사인의 판단이 어디에 남아 있어야 하는지를 설계 단계에서 명확히 한다.** 이것이 본 문서 전체를 관통하는 대원칙 — 숫자는 결정적 코드가, 문장은 LLM이, 결론은 감사인이 만든다 — 의 배경이다.

## 3. 업무 프로세스 분석 (As-Is)

대상 프로세스: 계정 사이클(매출·매출채권 기준) 감사 실무의 자료 수집~조서화 전 과정.

> **소요시간 비중 추정의 가정**: 아래 비중은 자산 1천억 원 미만 비상장사 감사에서 담당 감사인 1인이 매출·매출채권 사이클을 수행하는 경우를 상정한, 작성자의 실무 경험 기반 **추정치**이다. 표본조사·통계에 근거한 수치가 아니며, 회사 규모·자료 품질에 따라 편차가 크다.

| 단계 | 수행 내용 | Pain Point | 시간 비중(추정) | 병목 원인 |
|---|---|---|---|---|
| ① PBC 요청 | 계정별 요청자료 리스트 작성·발송 | 요청 항목–감사주장–수행절차의 연결이 담당자 머릿속에만 존재. 누락 시 재요청 왕복 | 5% | 요청 리스트의 비표준화 |
| ② 수취·분류 | 파일 수취, 종류 식별, 폴더 정리 | 파일명·형식 제각각, 무엇이 왔고 안 왔는지 수기 추적 | 10% | 수취 상태 관리의 부재 |
| ③ 정리·검증 | 열 이름·날짜·금액 형식 통일, 전표 단위 차대변 대사, 명세서 합계 vs 시산표 잔액 대사 | 회사마다 다른 원장 포맷을 매번 수작업 정리. 중복·결측·기간 외 전표를 눈으로 탐지 | 25% | 표준 스키마 부재, 검증의 비체계성 |
| ④ 분석적검토 | 전기 대비 증감, 월별 추이, 거래처별 분석, 이상 변동 식별 | 피벗·수식 반복 작성. 어디를 먼저 봐야 하는지 우선순위 없이 전체를 훑음 | 25% | 계산과 판단이 분리되지 않은 채 뒤섞임 |
| ⑤ 질의·회신 정리 | 이상 변동에 대한 회사 질의, 회신·증빙과 숫자 대조 | 질의 문안 작성 반복, 회신 내용과 숫자의 정합성 재확인 | 15% | 질의 문안·근거 수치의 수기 결합 |
| ⑥ 조서화 | 수행절차·변동내역·회사설명·잠정결론 문서화 | 동일 구조 조서의 반복 작성, 조서 내 숫자와 원천 데이터의 불일치 리스크 | 20% | 템플릿 부재, 수치 전기(轉記) 오류 |

> **주**: 외부조회(조회서 발송·회신 대사, 감사기준서 505)는 매출채권 감사의 핵심 실증절차이나, 회신 문서가 비정형 입력이어서 본 PoC의 분석 범위에서 제외하였다(로드맵 3단계). 상기 비중은 조회 절차를 제외한 기준이다.

**종합**: 판단 이전 단계(②③④의 기계적 부분 + ⑥의 수치 전기)가 전체의 절반 이상을 점유하는 것으로 추정된다. 특히 ③·④는 입력이 정형 데이터이고 처리 규칙이 명확하여 결정적 코드로 대체 가능성이 높고, ⑤·⑥의 문장 작성은 수치가 통제되는 전제 하에 LLM 지원 가능성이 높다.

## 4. 개선 과제 도출

### 4-1. Pain Point별 자동화 적용성 평가

| Pain Point | 자동화 유형 | 적용성 | 판단 근거 |
|---|---|---|---|
| 요청 리스트 비표준 (①) | 결정적 코드(계정–감사주장–절차 매핑 테이블) + LLM(안내 문구만) + **HITL(발송 전 검토)** | 상 | 요청항목–감사주장–절차의 연결이 정형 지식이라 테이블화 용이 |
| 수취 상태 수기 추적 (②) | 결정적 코드(헤더 시그니처 자료종류 판별, 이벤트 기반 상태 갱신) | 상 | 규칙 판별로 충분, LLM 개입 불필요 |
| 원장 포맷 비표준 (③) | 결정적 코드(매핑 사전·퍼지 매칭) + LLM(미지 헤더 후보 제안) + **HITL(매핑 확정)** | 상 | 규칙 처리율이 높고, 예외만 LLM 제안 후 감사인이 확정하면 오분류 리스크 통제 가능 |
| 중복·결측·차대변·잔액 대사 (③) | 결정적 코드 100% | 상 | 검증 룰이 명확한 전형적 규칙 검사. LLM 개입 불필요 |
| 증감·추이·이상치 식별 (④) | 결정적 코드(계산·스코어링) + LLM(후보별 해석 문장) | 상 | 계산은 pandas로 전수 처리, 우선순위는 규칙 가중치, LLM은 후보 제시 보조까지만 |
| 질의 문안 반복 작성 (⑤) | 결정적 코드(수치 주입·사후 대조) + LLM(문장) + **HITL(발송 전 검토)** | 중상 | 숫자를 코드가 주입하고 사후 검증하면 문장 생성 위임 가능 |
| 조서 수치 전기 오류·반복 작성 (⑥) | 결정적 코드(템플릿·수치 주입·검증기) + LLM(서술 문단) + **HITL(잠정결론 승인)** | 중상 | 결론 문구는 단정을 배제한 표준 문안으로 고정, 승인 전 수치 전수 대조 필수 |
| 조회서 대사, 주석 상호참조 | 결정적 코드(대사) + LLM(회신 문서 해석) 예상 | 중 | 비정형 입력 의존도가 높아 로드맵 단계에서 검토 |

> **적용성 등급 기준** — 상: 규칙 처리 가능 비율이 높고 구현 리스크 낮음 / 중상: 수치 통제 전제 하에 LLM 위임 가능 / 중: 비정형 입력 의존으로 추가 검토 필요.

### 4-2. 선정 과제와 논리

PoC에서는 다음 4개 과제를 선정한다.

| # | 과제 | 선정 논리 |
|---|---|---|
| 과제1 | **자료 표준화·검증 자동화** (열 매핑, 검증 룰 5종, 명세서–시산표 잔액 대사) | As-Is 비중 최대 구간(③). 규칙 기반이라 구현 리스크가 낮고, 이후 모든 분석의 데이터 품질 전제가 됨 |
| 과제2 | **분석적검토 지원** (전기 대비 증감·월별 추이·기말 집중도·신규/휴면 거래처의 계산·집계와 확인 필요 후보 스코어링 자동화) | 판단 이전의 계산·집계를 전수 자동화하고, 감사인의 시선을 "어디를 먼저 볼 것인가"로 이동시키는 핵심 가치. 절차의 수행 주체는 감사인이며 도구는 계산·후보 제시를 지원 |
| 과제3 | **문서화 지원** (질의 문안·조서 초안 생성, 수치는 코드 주입) | ⑤·⑥의 반복 작성 부담을 줄이되, 수치 통제와 승인 게이트로 품질 리스크를 차단 |
| 과제4 | **품질 통제 장치** (Review 검증기 — 생성문 수치 전수 대조, 평가 하네스) | 과제3이 신뢰를 얻기 위한 필수 전제. LLM 산출물의 숫자를 기계적으로 재검증하는 통제가 없으면 과제3은 도입 불가 |

- Pain Point ①(요청 리스트 비표준)·②(수취 상태 추적)는 별도 과제로 세우지 않고 **과제1의 전·후단 지원 기능으로 흡수**한다(요구사항 FR-01·FR-02·FR-14로 반영).
- 과제 간 관계: 과제1(데이터 품질) → 과제2(분석) → 과제3(문서화)이 파이프라인을 이루고, 과제4가 파이프라인 전체를 횡단 통제한다. 조회서 대사·주석검토는 입력이 비정형이어서 PoC의 검증 목적(역할 분리 구조의 실효성 확인)에 부합하지 않아 로드맵으로 이관한다.

## 5. 프로젝트 목표 및 범위

### 5-1. 목표
- 매출·매출채권 사이클 1개를 대상으로, "PBC 요청 → 업로드·매핑 → 검증 → 분석적검토 → 질의 → 조서 초안 → 수치 검증 → 승인 → Export"가 하나의 샘플 시나리오로 관통되는 동작 PoC를 1일 내 구축한다.
- 결정적 코드 / LLM / 감사인의 3자 역할 분리 구조가 실제 화면 흐름에서 성립함을 검증한다.

### 5-2. PoC 범위 (In)

| 구분 | 내용 |
|---|---|
| 대상 계정 | 매출, 매출채권 (FY2025 가상 회사 시나리오) |

## 6. 서비스 개요 (To-Be)

### 6-1. 파이프라인

```
[PBC 요청]      계정–감사주장–절차 매핑 테이블(코드) + 안내 문구(LLM) → 감사인 발송 전 검토
    ↓
[업로드·분류]   헤더 시그니처로 자료종류 판별 (코드, 실패 시 수동 지정 폴백)
    ↓
[열 매핑]       매핑 사전 + 퍼지 매칭(코드), 미지 헤더만 LLM이 confidence 포함 후보 제안
    ↓
◆ 게이트 1 ◆   감사인 매핑 확인·수정 후 확정 → 확정 매핑 JSON 저장
    ↓
[검증]          결정적 룰 5종 자동 실행 (LLM 미개입):
                중복 / 결측 / 전표번호 단위 차대변 일치 / 기간 외 전표 / 명세서–시산표 잔액 대사
    ↓
[분석적검토]    증감·월별 추이·기말 집중도·신규/휴면 전수 계산 + 규칙 스코어링(코드)
                → "확인 필요 후보 Top 5" (근거 수치·감사주장 태그 포함)
    │
    ├─ 분기: [질의 문안 생성]  숫자는 코드 주입 + 문장만 LLM
    │        → 생성 직후 Review 검증기 수치 대조 (불일치 시 표시 차단)
    │        → ◆ 게이트 2 ◆ 감사인 검토 후 발송 (자동 발송 없음)
    │
    └─ 본류: [조서 초안]  수치·표 코드 주입 + 서술 문단만 LLM + DRAFT 워터마크
             → [Review 검증기]  수치 전수 추출·원천값 대조 (불일치 시 승인 차단)
             → ◆ 게이트 3 ◆ 감사인 잠정결론 승인
             → [Export]  xlsx 3시트 + Markdown
```

### 6-2. HITL 승인 게이트 3개

| 게이트 | 위치 | 감사인의 결정 |
|---|---|---|
| 게이트 1 | 열 매핑 확정 | LLM이 제안한 매핑 후보(confidence 포함)를 확인·수정 후 확정. 확정 매핑은 JSON 저장(재현성) |
| 게이트 2 | 질의 문안 발송 전 | 생성 문안을 편집·검토. "감사인 검토 후 발송" 라벨 고정(자동 발송 기능 없음) |
| 게이트 3 | 조서 잠정결론 승인 | Review 검증기 통과를 전제로 승인. 불일치 1건이라도 있으면 승인 버튼 비활성화 |

### 6-3. As-Is 대비 달라지는 점

| 구분 | As-Is | To-Be |

## 8. 시스템 구성

### 8-1. 모듈 구조

Streamlit 단일 앱 + 3계층 모듈. 본 PoC 구현물은 자율 루프를 도는 에이전트가 아니라 **도구 단위로 분리된 함수 파이프라인 + 3개의 인간 승인 게이트**이며, 문서와 코드 모두 이를 그대로 서술한다. 각 단계가 독립 모듈로 분리되어 있어 향후 에이전트 오케스트레이션으로 확장 가능하다는 점이 구조적 가치다.

```
auditpilot/
├── app.py            # Streamlit (탭: 요청→업로드→검증→분석→조서→Harness)
├── core/             # 결정적 코드 계층 (pandas·openpyxl, LLM 미접촉)
│   ├── ingest.py     # 파일 로드·자료종류 판별      ├── analytics.py  # 증감·스코어링
│   ├── mapping.py    # 매핑 사전·퍼지 매칭          ├── workpaper.py  # 조서 수치 주입
│   ├── validate.py   # 검증 룰 5종                  ├── reviewer.py   # 수치 전수 대조
│   └── export.py     # xlsx·md
├── llm/              # client.py(추상화) · prompts.py("숫자 생성 금지" 명시) · cache.py(폴백)
├── harness/          # golden/(10케이스) · eval.py(결정적 채점) · results/
├── fixtures/         # 캐시 응답, 수치 불일치 조서 검증용 표본
└── data/make_sample.py  # 시드 고정 샘플 생성
```

### 8-2. 책임 분리

| 단계 | 결정적 코드 | LLM | 감사인 (HITL) |
|---|---|---|---|
| 요청 | 계정–감사주장–절차 매핑 테이블 | 안내 문구 문장만 | 발송 전 검토 |
| 매핑 | 사전+퍼지 매칭, 날짜·금액 파싱 | 미지 헤더 매핑 후보 제안(JSON) | **매핑 확정 (게이트 1)** |
| 검증 | 룰 5종 전부 | 없음 | 예외 항목 확인 |
| 분석 | 모든 계산·스코어링·정렬 | 후보별 한 줄 해석(계산값 주입) | 후보 채택 여부 |
| 질의 | 숫자 주입, 사후 전수 대조 | 주변 문장 작성 | **발송 전 검토 (게이트 2)** |
| 조서 | 수치·표 주입, 근거 각주, 템플릿 | 서술 문단만 | **잠정결론 승인 (게이트 3)** |
| 검증기 | 수치 추출·대조·승인 차단 | 없음 | 불일치 시 최종 판단 |

### 8-3. LLM Provider 추상화
- `LLMClient` 인터페이스 아래 OpenAI 호환 구현체 단일화(클라우드/Ollama 공용, base_url 전환). `.env` 1줄로 provider 교체.
- 목적: 클라이언트 데이터의 외부 전송이 허용되지 않는 환경(감사 대상회사 정보보안 정책, 법인 내부 통제)에서 온프레미스 LLM으로 동일 파이프라인을 구동하기 위함. 이하 데이터 보안 관련 서술(10-2, 12장)은 본 절을 근거로 한다.

### 8-4. 품질 통제 장치

**① Review 검증기 (FR-09)**
1. 생성문에서 정규식으로 수치 토큰 전수 추출 — 쉼표 포함 금액, 백분율, 건수 패턴
2. 정규화(쉼표·단위 제거, 부호 처리) 후 원천 계산값 딕셔너리(각 값에 근거 행 참조 포함)와 대조, 허용 오차 0
3. 항목별 일치/불일치 체크리스트 + 통과 배지. 불일치 1건이라도 존재하면 해당 문장 하이라이트 + 승인 차단
4. 동일 함수를 조서 검증 외에 질의 문안 사후 검증과 하네스 "숫자 무결성" 채점에 재사용한다(총 3개 사용 지점)

**② 평가 하네스 (FR-10)** — 정성 평가·주관 점수를 배제한 결정적 채점만 사용

| 지표 | 내용 |
|---|---|
| 매핑 정확도 | 제안 매핑 vs 정답 매핑 필드 일치율 |
| 숫자 무결성 통과율 | 생성문 수치와 주입값의 전수 일치율 (hallucination의 결정적 대리 지표) |
| 출력 스키마 준수율 | JSON 파싱 성공 + 필수 키 존재 |
| 금지 표현 위반 | 단정적 결론 어구 포함 여부 — NFR-04의 기계적 검사 |
| 지연·비용 | p50 지연시간, 토큰 기반 추정 비용 |

- 대상: 클라우드 API 1종 + Ollama 로컬 1종. 사전 실행 결과 JSON을 저장소에 기록하고 정적 표로 전시.
- **한계 명시**: 본 평가는 골든셋 10케이스의 스모크 수준 비교이며 통계적 유의성을 주장하지 않는다. 확장 시 케이스 수 증대, 재현율 기반 이상후보 평가, 프롬프트 버전별 회귀 테스트를 로드맵에 둔다.

## 13. 한계 및 유의사항

### 13-1. 이 도구가 하지 않는 것
- **감사 판단을 대체하지 않는다.** 확인 필요 후보는 우선순위 제안일 뿐 왜곡표시의 판정이 아니며, 후보에 오르지 않은 항목이 문제없다는 의미도 아니다. 위험 평가, 증거의 충분성·적합성 판단, 결론 도출은 전적으로 감사인의 몫이다.
- **단정적 결론을 출력하지 않는다.** 잠정결론은 단정적 판정을 배제한 표준 문안(감사인 승인 전까지 잠정 표시)으로 고정되며, 시스템·LLM 어느 쪽도 "적정하다" 류의 판정 어구를 생성하지 않는다(하네스 금지 표현 검사로 기계적 통제).
- **자동 발송·자동 확정이 없다.** 질의 발송, 매핑 확정, 조서 승인은 모두 감사인의 명시적 행위를 요구한다.

### 13-2. 감사기준 관점 유의점
- **사용하는 정보의 신뢰성(감사기준서 500)**: 본 도구의 검증 룰은 수취한 파일 내부·파일 간 정합성 검사에 한정된다. 분석에 사용하는 원천 데이터(원장·시산표)가 회사 회계시스템으로부터 완전하고 정확하게 생성·추출되었는지에 대한 평가는 감사인이 별도로 수행해야 하며, 검증 리포트의 통과 표시는 이를 대체하지 않는다. 같은 이유로 본 도구의 검증 룰은 기록 누락(완전성)에 대한 감사증거를 제공하지 않는다.
- **분석적절차의 위상(감사기준서 520)**: 본 도구의 분석 산출물은 위험평가절차 및 전반적 검토 단계의 분석 지원에 적합하다. 실증적 분석적절차로 활용하려면 기대치 수립·허용가능한 차이 설정·데이터 신뢰성 평가 등 기준서 520의 요구사항을 감사인이 별도로 충족해야 하며, 본 도구의 전기 대비 증감·임계치 스코어링은 그 요건을 자동으로 충족하는 설계가 아니다.
- **산출물의 증거 지위**: 본 도구의 산출물은 그 자체로 감사증거가 아니며, 감사인이 이를 검토·평가하여 수행한 절차를 통해 감사증거의 일부가 될 수 있다. 위험평가절차·실증절차로서의 활용 여부와 범위는 감사인이 감사기준에 따라 결정한다.
- **문서화 책임(감사기준서 230)**: 조서 초안의 사용은 문서화 책임(수행 절차·증거·결론의 기록)을 감사인으로부터 이전시키지 않는다. DRAFT 워터마크와 승인 게이트는 이를 구조적으로 상기시키기 위한 장치다.
- **임계치의 설정 주체**: 중요성 금액·수행중요성·명백히 사소한 왜곡표시 기준 등 스코어링 임계치는 감사인의 판단으로 설정되어야 하며, PoC의 기본값은 시연용 가정치다.
- **LLM 산출물 평가의 수준**: 현재 평가는 스모크 수준이다(8-4 한계 참조). 실무 적용 확대 전 1단계 로드맵의 평가 확충이 선행되어야 한다.
